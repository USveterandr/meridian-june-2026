### 1. Deployment pipeline is split-brain
The domain `investwithmeridian.com` is served by the Pages project **`meridian`** (direct upload only, no git). The git-connected project **`meridian-mobile`** builds from GitHub but serves only `meridian-mobile.pages.dev`. Consequences: git pushes never reach the live site, deploys are manual zip uploads, and the repo and production can silently diverge (today's commit `8cbbf27` is still not pushed to GitHub â€” production is ahead of the remote repo).

**Fix:** push `main` to GitHub, then in the dashboard move the `investwithmeridian.com` custom domain from `meridian` to `meridian-mobile`. After that, every push deploys the domain â€” including the per-listing OG-tag Pages Function, which direct uploads cannot ship.

### 2. `og-image.jpg` does not exist
Every page (index.html, seo.ts, sitemap) references `https://investwithmeridian.com/og-image.jpg`, but the file is not in the build â€” the URL returns the HTML shell. **Every social share of the site shows a broken/missing preview image.** Fix: add a 1200Ã—630 branded JPEG at `web/public/og-image.jpg`.

### 3. Stripe checkout is broken by plan-ID mismatch (pre-existing, still open)

Choosing Stripe checkout for `pro`, `brokerage`, or `investor` fails with "No Stripe price configured." Reconcile IDs and create real Stripe prices before enabling paid checkout.

### 4. No listing has coordinates â€” maps are dead sitewide
All 35 properties have `latitude = NULL`. The map section never renders on any listing, and the search map shows the "add VITE_GOOGLE_MAPS_KEY" fallback. Two missing secrets: `GOOGLE_MAPS_SERVER_KEY` (Worker, enables geocoding on create) and `VITE_GOOGLE_MAPS_KEY` (web build). After setting the server key, backfill existing listings with a one-time geocoding pass.

### 5. `city` column holds neighborhoods and avenues
Values include "Av. Independencia", "Av. Anacaona", "Mirador Norte", "La Julia", "Marbella, Juan Dolio". Effects: the homepage "Explore by City" grid finds almost nothing (â‰ˆ20 Santo Domingo listings don't match "Santo Domingo"), city search is unreliable, and the platform looks emptier than it is. Fix: normalize into `city` + `sector` columns (most of these are Santo Domingo sectors), and make the importer map sectors â†’ city.

### 6. Plan listing-limit bypass via PATCH
`POST /api/properties` enforces the plan's active-listing limit, but `PATCH /api/properties/:id` allows flipping `status` back to `active` with no limit check. A FREE-plan user can mark a listing inactive, create a new one, then re-activate the old one â€” unlimited active listings. Fix: run the same limit check in PATCH when the update sets `status = 'active'`.

### 7. Phone number disclosure via `/api/users/:id`
Any authenticated user can fetch any user ID (sequential integers) and get first/last name, role, and **phone**. Buyers/renters never consented to exposing their phone. Fix: only return `phone` when the target user owns at least one active listing (or has an agent-type role).

### 9. Rate limiting is per-isolate only
`middleware/rateLimit.ts` is documented as first-line defense; the real guarantee should be a Cloudflare WAF rule (`POST /api/auth/*`, 10 req/5 min/IP). I can't verify WAF config via API â€” confirm it exists in the dashboard, or credential-stuffing protection is soft.

### 10. JWT/session hardening
Tokens live in localStorage (exfiltratable by any XSS; the new CSP reduces this risk substantially), expire after 24 h, and there is no server-side revocation or refresh rotation. Acceptable for current scale; consider httpOnly-cookie sessions or short-lived tokens + refresh when payments go live.

### 11. `www.investwithmeridian.com` still dead + stale domain binding
No DNS record for `www` (the new `_redirects` rule handles it once the CNAME/custom domain exists). Also `investwithmeridian.isaac-trinidad.com` on meridian-mobile still 404s (inactive binding).
